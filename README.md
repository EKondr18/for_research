# PDF-семантический поиск и вопросы-ответы

Веб-приложение: задаёте вопрос на естественном языке — получаете ответ,
сгенерированный на основе содержимого PDF-документов из папки Google Drive,
плюс список источников (файл, страница, прямая ссылка на файл в Drive).

Полностью без Docker. Без платных API. Стек:

- **Frontend**: Next.js, деплой на **Vercel**
- **Backend**: FastAPI, деплой на **Render** как нативное Python-приложение (без Docker)
- **Векторная база**: **Qdrant Cloud** (free tier)
- **LLM для ответов**: **Groq API** (free tier)
- **Хранилище документов**: папка **Google Drive**, расшаренная «у кого есть ссылка», доступ по простому Google API Key (без OAuth, без Service Account)

---

## ⚠️ ВАЖНОЕ ПРЕДУПРЕЖДЕНИЕ О БЕЗОПАСНОСТИ

Способ доступа к Google Drive в этом проекте — **публичная ссылка «Все, у кого
есть ссылка»** плюс простой API Key. Это означает:

- **Любой человек, у которого окажется ссылка на папку** (переслали в чате,
  скопировали в письме, попала в лог, сохранилась в истории браузера и т.п.),
  **сможет открыть и скачать все файлы в этой папке**, без какой-либо
  авторизации в вашей организации.
- Google API Key, используемый в этом проекте, даёт доступ **только** к
  контенту, уже расшаренному по ссылке — сам по себе он не «взламывает»
  приватные файлы. Но именно поэтому уровень защиты документов равен
  **уровню защиты самой ссылки**, а не вашей корпоративной системы доступа.
- Ссылка на backend (`/ask`) и сами эндпоинты API также по умолчанию не
  защищены авторизацией пользователей (см. раздел «Продакшн-соображения»).

**Если вы планируете загружать в эту папку служебные документы — протоколы,
отчёты о нарушениях, регуляторные материалы, договоры, персональные данные
и т.п. — обязательно согласуйте такой способ хранения с ИБ/руководством
до того, как туда попадут реальные рабочие документы.** Для чувствительных
документов эта архитектура (публичная ссылка + API Key) не рекомендуется —
рассмотрите OAuth/Service Account с ограниченным доступом или отдельное
защищённое хранилище (это потребует доработки кода).

## Архитектура

```
Пользователь → Vercel (Next.js фронт)
                   │  POST /ask { question }
                   ▼
              Render (FastAPI backend)
                   │
        ┌──────────┼──────────────┐
        ▼          ▼              ▼
  Google Drive   Qdrant Cloud   Groq API
   (файлы)     (векторный поиск) (генерация ответа)
```

- `POST /reindex` — обходит папку Drive, скачивает новые/изменённые PDF,
  извлекает текст, режет на чанки, считает эмбеддинги, пишет в Qdrant.
- `POST /ask` — вопрос → поиск релевантных чанков в Qdrant → контекст в Groq
  → ответ + список источников.
- `GET /health` — проверяет доступность Google Drive API, Qdrant, Groq.

## 1. Настройка Google Drive (API Key, без OAuth)

1. Загрузите PDF-документы в отдельную папку на Google Drive.
2. ПКМ по папке → **Доступ** (Share) → **Общий доступ** → измените на
   **«Все, у кого есть ссылка»** (Anyone with the link) с ролью
   **«Читатель»** (Viewer). Скопируйте ссылку — она выглядит так:
   `https://drive.google.com/drive/folders/1AbCдефGhIJKlmnoPQRstuVWxyz`.
   Всё, что после `/folders/` — это **Folder ID** (`1AbCдефGhIJKlmnoPQRstuVWxyz`),
   он пойдёт в переменную `GOOGLE_DRIVE_FOLDER_ID`.
3. Получите API Key в Google Cloud Console:
   - Зайдите на https://console.cloud.google.com/
   - Создайте новый проект (или выберите существующий).
   - В меню **APIs & Services → Library** найдите **Google Drive API** и
     нажмите **Enable** (включить). Без этого шага все запросы будут
     возвращать 403.
   - Перейдите в **APIs & Services → Credentials → Create Credentials →
     API Key**. Скопируйте ключ — это `GOOGLE_DRIVE_API_KEY`.
   - Рекомендуется (необязательно) ограничить ключ по **API restrictions**
     → разрешить только **Google Drive API**, чтобы ключ нельзя было
     использовать для других Google-сервисов, если он утечёт.
4. Проверка: откройте в браузере в приватном окне (без входа в аккаунт Google)
   `https://www.googleapis.com/drive/v3/files?q='FOLDER_ID'+in+parents&key=API_KEY`
   — если видите список файлов в JSON, всё настроено верно. Если видите
   ошибку 403 — папка не расшарена как «у кого есть ссылка», либо Drive API
   не включён для проекта.

## 2. Настройка Qdrant Cloud (free tier)

1. Зарегистрируйтесь на https://cloud.qdrant.io/
2. Создайте новый **Free tier cluster** (обычно 1GB, этого достаточно для
   нескольких тысяч страниц PDF).
3. В настройках кластера скопируйте **Cluster URL** (`QDRANT_URL`, вида
   `https://xxxxxxx.eu-central-1-0.aws.cloud.qdrant.io:6333`).
4. Создайте **API Key** для кластера (Data Access Control → Create) —
   это `QDRANT_API_KEY`.

Коллекции (`pdf_chunks`, `documents_meta`) backend создаёт автоматически при
первом вызове `/reindex` или `/health` — вручную ничего создавать не нужно.

## 3. Настройка Groq API (free tier)

1. Зарегистрируйтесь на https://console.groq.com/ (можно через Google-аккаунт,
   карта не требуется).
2. В разделе **API Keys** создайте ключ — это `GROQ_API_KEY`.
3. Модель по умолчанию в этом проекте — **`openai/gpt-oss-120b`**
   (128K контекст). Это актуальная бесплатная модель на момент написания
   (июль 2026): ранее рекомендованная `llama-3.3-70b-versatile` была
   деприкейтнута Groq в июне 2026. Если понадобится другая модель (например,
   `qwen/qwen3-32b` — тоже неплохо работает с русским), задайте её через
   переменную окружения `GROQ_MODEL`. Актуальный список моделей и лимитов
   всегда проверяйте на https://console.groq.com/docs/models — Groq иногда
   меняет/деприкейтит модели без предупреждения за недели вперёд.

## 4. Деплой backend на Render (без Docker)

1. Запушьте этот репозиторий на GitHub.
2. На https://dashboard.render.com/ → **New → Web Service**.
3. Подключите репозиторий, укажите:
   - **Root Directory**: `backend`
   - **Runtime**: `Python 3` (Render сам подхватит `runtime.txt` с версией 3.11)
   - **Build Command**: `pip install -r requirements.txt`
   - **Start Command**: `uvicorn main:app --host 0.0.0.0 --port $PORT`
   - **Instance Type**: Free
4. В разделе **Environment** добавьте переменные:
   ```
   GOOGLE_DRIVE_API_KEY=...
   GOOGLE_DRIVE_FOLDER_ID=...
   QDRANT_URL=...
   QDRANT_API_KEY=...
   GROQ_API_KEY=...
   GROQ_MODEL=openai/gpt-oss-120b        # необязательно, это значение по умолчанию
   REINDEX_TOKEN=какой-нибудь-секрет      # рекомендуется, см. ниже
   CORS_ALLOW_ORIGINS=https://ваш-домен.vercel.app
   ```
5. Deploy. После первого успешного деплоя проверьте `GET /health` — все три
   сервиса должны быть `"ok"`.
6. Запустите первую индексацию:
   ```
   curl -X POST https://ваш-backend.onrender.com/reindex \
     -H "X-Reindex-Token: какой-нибудь-секрет"
   ```
   Индексация может занять время на первом запуске (Render впервые скачивает
   embedding-модель) — см. раздел про лимиты ниже.

**Про `REINDEX_TOKEN`**: `/reindex` скачивает файлы и пересчитывает
эмбеддинги — это дорогая по ресурсам операция. Если `REINDEX_TOKEN` не
задан, эндпоинт открыт для любого, кто узнает URL backend'а. Рекомендуется
задать секрет и передавать его в заголовке `X-Reindex-Token` при вызове
(вручную, через cron, GitHub Actions и т.п.).

## 5. Деплой frontend на Vercel

1. На https://vercel.com/ → **Add New → Project** → импортируйте тот же
   репозиторий.
2. **Root Directory**: `frontend`. Vercel сам определит Next.js и настроит
   build/start команды — ничего менять не нужно.
3. В **Environment Variables** добавьте:
   ```
   NEXT_PUBLIC_API_URL=https://ваш-backend.onrender.com
   ```
4. Deploy. Откройте выданный Vercel URL — форма вопроса должна работать.

## Локальный запуск (без Docker)

### Backend
```bash
cd backend
python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
cp .env.example .env   # заполните значениями и экспортируйте в shell
export $(grep -v '^#' .env | xargs)
uvicorn main:app --reload
```

### Frontend
```bash
cd frontend
npm install
cp .env.local.example .env.local   # укажите NEXT_PUBLIC_API_URL=http://localhost:8000
npm run dev
```

## Лимиты бесплатных тиров — что важно понимать

| Сервис | Лимит free tier | Что это значит для вас |
|---|---|---|
| **Render** | Web Service засыпает после ~15 минут без запросов; первый запрос после сна отвечает 30-60+ секунд ("cold start") | Первый вопрос пользователя после простоя будет долгим. Фронтенд показывает это в сообщении об ошибке/задержке. Можно (опционально) пинговать `/health` внешним cron каждые ~10 минут, чтобы снизить частоту засыпания — но это не отключает лимит бесплатного тира, а просто держит сервис "тёплым" в рамках допустимого использования. |
| **Qdrant Cloud free** | Обычно 1GB на кластер (лимиты меняются, проверяйте в дашборде) | На объём ~1GB помещается порядка нескольких десятков тысяч чанков (зависит от размера чанка). Если растёте — потребуется платный план или отдельный self-hosted Qdrant. |
| **Groq free tier** | Лимиты по RPM/RPD/TPM/TPD, зависят от модели (см. https://console.groq.com/docs/rate-limits) | При интенсивном использовании возможны 429 ошибки — `/ask` в этом случае вернёт 502 с текстом ошибки Groq. Для продакшн-нагрузки нужен платный план Groq. |
| **Google Drive API** | Квоты на количество запросов в сутки/100 секунд (обычно щедрые для чтения, но не бесконечные) | `/reindex` обходит все файлы в папке при каждом вызове (но скачивает только новые/изменённые) — не запускайте reindex слишком часто/параллельно. |

## Как переиндексация решает, что обновлять

Backend хранит в отдельной Qdrant-коллекции (`documents_meta`) для каждого
проиндексированного файла его `drive_file_id` и `modifiedTime`. При каждом
`POST /reindex`:

1. Получает актуальный список PDF в папке Drive (id, имя, modifiedTime).
2. Сравнивает с уже сохранёнными `modifiedTime`.
3. Скачивает и переиндексирует **только** новые файлы и файлы, у которых
   `modifiedTime` изменился с прошлого раза. Остальные пропускает.
4. Файлы без текстового слоя (сканы без OCR) логируются как warning и
   пропускаются — их содержимое не попадёт в поиск, но сам факт наличия
   файла запоминается, чтобы не пытаться переиндексировать его на каждый
   вызов.

OCR на этом этапе не реализован — если у вас много сканированных документов
без текстового слоя, потребуется отдельный OCR-этап перед индексацией
(например, `pytesseract`), который в эту версию не входит.

## Продакшн-соображения (за рамками текущей реализации)

- `/ask` и `/reindex` не защищены пользовательской аутентификацией — любой,
  кто знает URL backend'а, может задавать вопросы (расходуя ваш Groq/Qdrant
  лимит). Для реального использования добавьте хотя бы простую защиту
  (API key на фронт↔бэк, Vercel/Render basic auth, rate limiting).
- Embedding-модель (`sentence-transformers/paraphrase-multilingual-MiniLM-L12-v2`,
  ~0.22GB) загружается через `fastembed` (ONNX Runtime, без PyTorch) —
  выбрана специально ради экономии RAM на Render free tier (512MB-1GB).
  Более тяжёлые модели (например, `intfloat/multilingual-e5-large`, ~2.2GB)
  дадут точнее эмбеддинги, но **с высокой вероятностью не влезут в память
  на бесплатном тарифе Render** — используйте их только на платных планах
  с большим RAM.
